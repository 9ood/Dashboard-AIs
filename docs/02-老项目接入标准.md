# 老项目接入标准 v1

这个文件是给已经存在的老项目用的。

目标：
- 不先大拆业务
- 先让项目有统一的门牌、开关和说明书
- 把历史问题分成“必须先修”和“可以后修”

## 1. 接入顺序

老项目接入统一系统，固定按这 4 步走：

### 第一步：定义边界

先回答 5 个问题：
- 这个项目到底是 `service`、`batch`、`queue-worker` 还是 `tool`
- 它真正的主入口文件是哪一个
- 它依赖哪些外部服务
- 它会读哪些目录
- 它会写哪些目录

### 第二步：补统一壳

不给业务大动刀，先补：
- `project.config.json`
- `scripts/test.*`
- `scripts/run.*`
- `scripts/stop.*`
- `.env.example`

### 第三步：修硬伤

只修会挡住接入的硬伤：
- 写死绝对路径
- 平台锁死
- 缺少启动入口
- 缺少日志目录
- 缺少最小测试动作

### 第四步：接入 dashboard

接入后能做到：
- 能显示
- 能控制
- 能看状态
- 能看日志

## 2. 必修项和缓修项

### 必修项

这些不修，不能进统一系统：
- 写死 mac 路径、Windows 路径
- 一个目录里放两份同名项目但没有主次说明
- 没有统一启动方式
- 没有最小测试方式
- 关键配置只存在个人本机，不存在 `.env.example`

### 缓修项

这些可以先带病接入：
- UI 风格老
- 代码风格不统一
- 文档不好看
- 内部模块划分不漂亮

原则：
- 先修“会炸系统”的
- 后修“看起来不舒服”的

## 3. 老项目必须补的统一文件

每个老项目接入前，至少补这几样：
- `project.config.json`
- `README.md`
- `.env.example`
- `scripts/test.ps1`
- `scripts/run.ps1`
- `scripts/stop.ps1`

如果要兼容 mac，再补：
- `scripts/test.sh`
- `scripts/run.sh`
- `scripts/stop.sh`

## 4. 平台适配规则

老项目以前很多是按 mac 写的，现在要改成通用。

统一做法：
- 业务逻辑改成平台无关
- 平台差异下沉到 `scripts/`
- `launchd` 和 `Task Scheduler` 都不写进业务主代码

推荐方式：
- Python 服务：统一从 `python -m` 或单一入口脚本启动
- Node 服务：统一从 `node xxx.js` 或 `npm run` 启动
- 定时任务：由系统层配置，不写死到项目核心代码

## 5. 目录重复的处理规则

如果一个项目里出现“又放了一份项目”，按下面规则处理：

### 情况 A：完全重复

动作：
- 先比对 remote、commit、关键文件差异
- 确认差异可迁移后，把有用改动合并到主项目
- 合并完成后，再删除重复目录

### 情况 B：不是重复，是依赖或示例

动作：
- 必须改名
- 必须在 README 写清楚用途
- 不能继续叫 `repo` 这种看不出身份的名字

### 重要规则

没有做完这三件事，不能删：
- 确认主版本
- 确认差异
- 确认回滚方式

## 6. 老项目接入验收标准

一个老项目接入完成，至少满足：
- dashboard 能识别它
- dashboard 能测试它
- dashboard 能启动它
- dashboard 能停止它
- dashboard 能看见最近一次状态

如果是网页服务，再加：
- dashboard 能打开入口地址
- dashboard 能做健康检查

## 7. 回滚标准

老项目接入一律小步走。

回滚顺序：
1. 先撤销 dashboard 的接入配置
2. 再撤销新增脚本
3. 最后才回滚业务代码

禁止：
- 一上来删老目录
- 一上来把旧入口替换掉
- 一上来做全量重命名

## 8. 推荐接入节奏

最稳的顺序：
- 先接最简单的批处理项目
- 再接网页服务
- 再接队列和调度类项目
- 最后处理平台耦合最重的项目

这样做的好处：
- 先拿到统一系统骨架
- 后面复杂项目有地方挂
- 历史问题不会一下子全爆出来
