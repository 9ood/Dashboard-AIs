# 新项目开发标准 v1

这个文件是以后新项目进入统一系统时必须遵守的规则。

目标很简单：
- 新项目一开始就按统一方法写
- 不再出现“项目能跑，但接不进 dashboard”的情况
- 不再把 mac、Windows、路径、日志、配置写死在业务代码里

## 1. 项目边界

一个项目只做一件事。

允许的项目类型：
- `service`：长期运行的网页服务或 API 服务
- `batch`：点一下就跑一轮的批处理任务
- `queue-worker`：持续消费队列的任务
- `tool`：外部工具的包装器

不允许：
- 一个项目里同时塞多个完全不同的业务
- 一个项目直接控制另一个项目的内部代码
- 一个项目把 dashboard 当作自己的依赖

## 2. 必须有的文件

每个新项目根目录，必须有这些文件：

- `project.config.json`
- `README.md`
- `.env.example`
- `scripts/`

推荐最小目录：

```text
project-name/
├── project.config.json
├── README.md
├── .env.example
├── scripts/
│   ├── test.ps1
│   ├── run.ps1
│   ├── stop.ps1
│   ├── test.sh
│   ├── run.sh
│   └── stop.sh
├── logs/
├── data/
├── output/
└── src/ 或 app/
```

## 3. project.config.json 统一规则

每个项目都必须声明这些信息：

```json
{
  "id": "music-news",
  "name": "每日音乐选题",
  "type": "batch",
  "entry": {
    "web": "",
    "health": ""
  },
  "runtime": {
    "windows": {
      "test": ["powershell", "-ExecutionPolicy", "Bypass", "-File", "scripts/test.ps1"],
      "run": ["powershell", "-ExecutionPolicy", "Bypass", "-File", "scripts/run.ps1"],
      "stop": ["powershell", "-ExecutionPolicy", "Bypass", "-File", "scripts/stop.ps1"]
    },
    "macos": {
      "test": ["bash", "scripts/test.sh"],
      "run": ["bash", "scripts/run.sh"],
      "stop": ["bash", "scripts/stop.sh"]
    }
  },
  "paths": {
    "logs": "logs",
    "data": "data",
    "output": "output"
  }
}
```

要求：
- `id` 只能有一个，后面不能乱改
- 所有路径都写相对路径
- 平台命令分开写，不要混在一起
- 没有的动作可以留空，但必须明确

## 4. 跨平台硬规则

以后所有项目都按“通用优先”来写。

必须做到：
- Python 路径处理用 `pathlib.Path` 或 `os.path`
- Node 路径处理用 `path.resolve`
- 配置从环境变量或配置文件读取
- 启动端口、主机、日志目录都可配置
- 平台相关脚本放进 `scripts/` 里

禁止：
- 代码里写死 `/Users/...`
- 代码里写死 `C:\\...` 或 `E:\\...`
- 代码里直接写 `launchd`、`task scheduler` 的业务逻辑
- 业务代码里直接判断“我只能在 mac 跑”

正确做法：
- 业务代码写通用
- 平台差异放到脚本层
- dashboard 只调脚本，不碰业务细节

## 5. 启动、停止、测试标准

每个项目最少要有 3 个动作：
- `test`
- `run`
- `stop`

规则：
- `test`：只检查项目能不能进入工作状态，不做危险动作
- `run`：真实运行
- `stop`：结束当前运行

如果项目是网页服务，还要补一个：
- `health`

健康检查要能回答：
- 服务在不在
- 端口是不是通的
- 最近一次启动是否成功

## 6. 日志标准

每个项目都必须把日志放在自己的 `logs/`。

最少分两类：
- `app.log`：业务日志
- `runtime.log`：启动、停止、异常退出日志

要求：
- 日志路径不能写死成系统绝对路径
- dashboard 只读取日志，不负责替你写业务日志

## 7. 数据和产物标准

必须分开：
- `data/`：输入数据
- `output/`：输出结果
- `logs/`：日志
- `tmp/`：临时文件

禁止：
- 输入、输出、日志全混在根目录
- 项目把别的项目目录当成自己的输出目录

## 8. 接入 dashboard 的最低要求

新项目要想接入 dashboard，至少满足：
- 有 `project.config.json`
- 有 `test/run/stop` 脚本
- 有清楚的日志目录
- 没有写死平台路径
- README 里写清楚启动方法和依赖

## 9. 开发完成前自检

每个新项目在接入前，都要过这张表：

- 能在 Windows 跑
- 能在 macOS 跑，或明确说明为什么只支持一种平台
- 不写死绝对路径
- 不依赖手工改源码才能启动
- 能输出最小测试结果
- 能被 dashboard 读取状态

## 10. 回滚规则

如果新项目接入失败：
- 先回滚 `project.config.json`
- 再回滚接入脚本
- 不直接删业务源码

原则：
- 接入失败，不影响旧系统
- 一次只处理一个项目
